# Spec 05 — Web Diagram Support

## Phase
Phase 3 (part of Web UI)

## Context
The web visualizer adds diagram rendering on top of Spec 04's HTML output. Two formats are supported:

- **GoAT** (`goat` fenced blocks): rendered server-side to inline SVG using the `bep/goat` library. No client-side work needed.
- **Mermaid** (`mermaid` fenced blocks): rendered client-side by the Mermaid JS library, which is bundled as an embedded asset — no CDN, fully offline.

This spec modifies the HTML rendering pipeline (`internal/renderer/html`) and the embedded web assets. It has no effect on the TUI rendering path (covered in Spec 03).

---

## Dependencies

| Library | Purpose |
|---|---|
| `github.com/bep/goat` | GoAT source → SVG (server-side) |
| Mermaid JS (bundled) | Mermaid source → diagram (client-side) |

Mermaid JS is downloaded at build time (or committed as a vendored asset) and embedded via `go:embed`. It is **not** fetched from a CDN at runtime.

---

## Package Layout

```
internal/
  renderer/
    html/
      renderer.go       extended: goldmark AST transform for goat + mermaid
      renderer_test.go

web/
  static/
    vendor/
      mermaid.min.js    bundled Mermaid JS (committed, not fetched at runtime)
    app.js              extended: Mermaid initialisation
```

---

## Approach

Rather than a pre-processing pass on raw Markdown bytes (as in Spec 03), the HTML renderer uses a **goldmark AST transformer**:

- A custom `goldmark.Transformer` inspects `FencedCodeBlock` nodes
- `goat` blocks are replaced with a `RawHTML` node containing the SVG output
- `mermaid` blocks are replaced with a `RawHTML` node containing `<div class="mermaid">…source…</div>` — Mermaid JS then processes these divs on page load

This approach is more robust than regex pre-processing and integrates cleanly with goldmark's extension model.

Note: the HTML renderer must be configured with `html.WithUnsafe(true)` to allow the injected raw HTML nodes to pass through to output. The content injected is always generated by our own code (GoAT SVG or escaped Mermaid source), never from user-supplied raw HTML, so this is safe.

## Tasks

---

### Task 1 — GoAT SVG transformer (goldmark extension)

**Goal:** A goldmark `Transformer` that replaces `goat` fenced code blocks with inline SVG in the rendered HTML output.

**Files:**
- `internal/renderer/html/goat_transform.go`
- `internal/renderer/html/goat_transform_test.go`

**Steps:**
1. Implement `goatTransformer` satisfying `util.Prioritized` (goldmark's `parser.ASTTransformer` interface)
2. In `Transform(doc *ast.Document, reader text.Reader, pc parser.Context)`: walk the AST, find `ast.FencedCodeBlock` nodes whose info string is `goat`
3. For each match: extract the block's raw text, call `goat.New(strings.NewReader(src)).BuildSVG()`, wrap the SVG in `<figure class="diagram goat">…</figure>`, replace the node with an `ast.RawHTML` node containing this HTML
4. If `BuildSVG` returns an error or produces empty output: replace with `<figure class="diagram goat error"><pre>` + html-escaped source + `</pre><figcaption>⚠ diagram could not be rendered</figcaption></figure>`
5. Register the transformer as a goldmark `Extender` so it is added via `goldmark.WithExtensions`
6. Unit tests: a document with a `goat` block produces HTML containing `<svg`; a document with no `goat` blocks is unchanged; a `goat` block with invalid/empty content produces the error figure

**Acceptance criteria:**
- `html.Render` on a document containing a `goat` block returns HTML with an inline `<svg` element
- The SVG is wrapped in `<figure class="diagram goat">`
- A document with no diagram blocks passes through with no SVG injected
- An empty `goat` block produces the error figure, not a panic or empty output
- `go test ./internal/renderer/html/... -run TestGoAT` passes

### Task 2 — Mermaid div transformer (goldmark extension)

**Goal:** A goldmark `Transformer` that replaces `mermaid` fenced code blocks with `<div class="mermaid">` elements that Mermaid JS will process on the client.

**Files:**
- `internal/renderer/html/mermaid_transform.go`
- `internal/renderer/html/mermaid_transform_test.go`

**Steps:**
1. Implement `mermaidTransformer` (same interface as Task 1)
2. For each `FencedCodeBlock` with info string `mermaid`: extract source, HTML-escape it, emit `<figure class="diagram mermaid"><div class="mermaid">` + escaped source + `</div></figure>`
3. Escaping is mandatory: the source is placed inside a div and Mermaid JS reads its `textContent`, so `<`, `>`, `&` must be escaped to prevent XSS if the `.md` file comes from an untrusted source
4. Unit tests: a `mermaid` block produces HTML containing `<div class="mermaid"`; source special characters (`<`, `>`, `&`) are escaped in the output; a document without mermaid blocks is unchanged

**Acceptance criteria:**
- `html.Render` on a document with a `mermaid` block returns HTML containing `<div class="mermaid"`
- `< > &` in the diagram source are HTML-escaped in the output
- A document with no mermaid blocks is unchanged
- `go test ./internal/renderer/html/... -run TestMermaid` passes

### Task 3 — Register transformers and enable unsafe HTML output

**Goal:** Wire both transformers into the goldmark instance in `html/renderer.go` and enable raw HTML passthrough.

**Files:**
- `internal/renderer/html/renderer.go`

**Steps:**
1. Add `goldmark.WithExtensions(&goatExtension{}, &mermaidExtension{})` to the goldmark constructor (the extenders registered in Tasks 1 and 2)
2. Change the goldmark HTML renderer option from the default to `html.WithUnsafe(true)` — required for `ast.RawHTML` nodes to appear in output
3. Existing unit tests from Spec 04 Task 1 must still pass — verify GFM tables, task lists, headings are unaffected

**Acceptance criteria:**
- `html.Render` with a `goat` block produces `<svg` in the output
- `html.Render` with a `mermaid` block produces `<div class="mermaid"` in the output
- All Spec 04 Task 1 acceptance criteria continue to pass
- `go test ./internal/renderer/html/...` passes

### Task 4 — Bundle Mermaid JS as an embedded asset

**Goal:** Mermaid JS is committed to the repository and embedded in the binary. No network fetch at build or runtime.

**Files:**
- `web/static/vendor/mermaid.min.js`
- `scripts/vendor-mermaid.sh` (one-off helper script, not run during build)
- `web/static/app.js` (extend)

**Steps:**
1. Download the latest stable Mermaid JS release (e.g. from the Mermaid GitHub releases page) manually and commit `mermaid.min.js` to `web/static/vendor/`; record the version and checksum in a comment at the top of the file or in a `web/static/vendor/README`
2. Add `scripts/vendor-mermaid.sh` as a documented one-off script for future updates — it should `curl` the file, verify the checksum, and copy it into place; it is not part of `go generate` or the build
3. Add `<script src="/static/vendor/mermaid.min.js"></script>` to `web/templates/base.html`, before `app.js`
4. In `app.js`, add after DOMContentLoaded: `mermaid.initialize({ startOnLoad: true, theme: 'default' })`; then call `mermaid.run()` after any dynamic content replacement (SSE reload, file fetch) to re-process newly injected `.mermaid` divs

**Acceptance criteria:**
- `go build ./...` succeeds and the resulting binary contains the Mermaid JS bytes (verify with `go tool nm` or by checking binary size delta)
- `GET /static/vendor/mermaid.min.js` returns HTTP 200 with the file contents
- The browser renders a `mermaid` block as a diagram with no network requests (verify with browser DevTools → Network tab, filter to "mermaid")
- After an SSE-triggered content reload, newly loaded `mermaid` blocks are also rendered (not left as raw text)

### Task 5 — CSS for diagram figures

**Goal:** Diagrams are visually distinct from regular content — centred, with a light border, and with the error state styled in a warning colour.

**Files:**
- `web/static/style.css`

**Steps:**
1. Add styles for `figure.diagram`: `display: block`, `margin: 1.5rem auto`, `text-align: center`, `border: 1px solid #e0e0e0`, `border-radius: 6px`, `padding: 1rem`, `overflow-x: auto`
2. `figure.diagram.error`: border and background in a muted warning tone (e.g. `#fff3cd` background, `#856404` border), `pre` inside styled as monospace small text
3. `figure.diagram svg`: `max-width: 100%`, `height: auto` (responsive)
4. Dark-mode media query (`prefers-color-scheme: dark`): adjust border and error colours appropriately

**Acceptance criteria:**
- A GoAT diagram is centred with a visible border in the browser
- An error figure (failed diagram) has a visually distinct warning style
- SVG diagrams do not overflow their container on narrow viewports (manual browser test at 400 px width)
- Dark-mode styles apply when the OS is in dark mode (manual browser test)
